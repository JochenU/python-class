(a) List growth

Python lists are highly optimized for performing append() operations. Each time a list grows, it grabs a larger chunk of memory than it actually needs with the expectation that more data will be added to the list later. If new items are added and space is available, the append() operation simply stores the item without allocating more memory.

Experiment with this feature of lists by using the sys.getsizeof() function on a list and appending a few more items.

>>> import sys
>>> items = []
>>> sys.getsizeof(items)
64
>>> items.append(1)
>>> sys.getsizeof(items)
96
>>> items.append(2)
>>> sys.getsizeof(items)    # Notice how the size does not increase
96
>>> items.append(3)
>>> sys.getsizeof(items)    # It still doesn't increase here
96
>>> items.append(4)
>>> sys.getsizeof(items)    # Not yet.
96
>>> items.append(5)
>>> sys.getsizeof(items)    # Notice the size has jumped
128
>>>

A list stores its items by reference. So, the memory required for each item is just a single pointer. On a 64-bit machine, a pointer is typically 8 bytes. However, if Python has been compiled for 32-bits, it might be 4 bytes and the numbers for the above example will be half of what’s shown.

(b) Bits and Bytes

Mutable containers such as lists can behave in other odd ways when it comes to memory. In part (a), you looked at how lists overallocate their space for appends, but let’s explore some other aspects of list allocation.

First, what happens if you take a slice?

>>> items
[1, 2, 3, 4, 5]
>>> sys.getsizeof(items)
128
>>> a = items[:]
>>> a
[1, 2, 3, 4, 5]
>>> sys.getsizeof(a)
104
>>>

Notice how the size of the slice is less than the original list, but it contains all of the same items. For certain list operations such as slices, if the size of the result can be exactly determined, the resulting list is set to exactly that size without any extra space. However, it doesn’t always work like you might expect. For example, creating a list copy with list() might increase the size of the result:

>>> sys.getsizeof(a)
104
>>> b = list(a)
>>> sys.getsizeof(b)
128
>>>

Here’s another interesting case. What happens if you split a string?

>>> s = '1,2'
>>> vals = s.split(',')
>>> vals
['1', '2']
>>> sys.getsizeof(vals)
160
>>>

Yikes! 160 bytes just to store the list created by splitting a 3 character string. Restart Python and try this simple experiment reading a file of data.

>>> # --- RESTART
>>> f = open('Data/ctabus.csv')
>>> rows = [line.split(',') for line in f]
>>> import os
>>> os.getpid()
5027
>>> # --- Look at memory use

Go look at the memory use of the resulting process and marvel at how large it is compared to similar work in Exercise 2.1. What about this variant that makes tuples instead?

>>> # --- RESTART
>>> f = open('Data/ctabus.csv')
>>> rows = [tuple(line.split(',')) for line in f]
>>> import os
>>> os.getpid()
5090
>>> # --- Look at memory use

Last, but not least, what is the memory footprint if you split with the csv module?

>>> # --- RESTART
>>> f = open('Data/ctabus.csv')
>>> import csv
>>> rows = list(csv.reader(f))
>>> import os
>>> os.getpid()
5102
>>> # --- Look at memory use

If there’s any take away from any of this, it’s that there can be great variability in the resource use of a Python program depending on how you construct your data. On the other hand, memory is often a lot cheaper than programmers. Keeping your code simple (even if it uses more memory) is often a sensible strategy.

(c) Dictionary/Class Growth

Python dictionaries (and classes) allow up to 5 values to be stored before their reserved memory increases four-fold. In Exercise 2.1, you wrote a program readrides.py that read data into a list of dictionaries and observed the memory use.

Suppose you wanted to modify that program to parse the date field into a more usable data structure. Suppose your first choice is to represent the date as a tuple like this:

def read_rides_as_dicts(filename):
    '''
    Read the bus ride data as a list of dicts
    '''
    rides = []
    f = open(filename)
    f_csv = csv.reader(f)
    headings = next(f_csv)
    for row in f_csv:
        route = row[0]
        datefields = row[1].split('/')
        date = (int(datefields[2]), int(datefields[0]), int(datefields[1]))   # (year, mon, day)
        daytype = row[2]
        numrides = int(row[3])
        ride = {
            'route': route,
            'date': date,
            'daytype': daytype,
            'numrides' : numrides
            }
        rides.append(ride)
    f.close()
    return rides

Suppose your other choice is to store the year, month, and day directly in the dictionary like this:

def read_rides_as_dicts(filename):
    '''
    Read the bus ride data as a list of dicts
    '''
    rides = []
    f = open(filename)
    f_csv = csv.reader(f)
    headings = next(f_csv)     # Skip headers
    for row in f_csv:
        route = row[0]
        datefields = row[1].split('/')
        daytype = row[2]
        numrides = int(row[3])
        ride = {
            'route': route,
            'year': int(datefields[2]),
            'month': int(datefields[0]),
            'day': int(datefields[1]),
            'daytype': daytype,
            'numrides' : numrides
            }
        rides.append(ride)
    f.close()
    return rides

Try each of these approaches and determine which one results in the most efficient use of memory.

Food for thought: If you are creating large numbers of records, you really should consider the use of tuples, named tuples, or classes that define __slots__.
